O que é o fluxo On-Behalf-Of?

  • Um middle-tier recebe um token do usuário
  • Troca esse token por outro, delegado, para um recurso downstream
  • O recurso final vê a identidade do usuário original

Delegado — não é application permission

Seguro — credencial confidencial no middle-tier

Auditável — token final identifica o usuário

OBO vs. outras abordagens

OBO (Delegado)App-OnlyToken direto
Identidade no Graph✅ UsuárioApp✅ Usuário
Respeita permissões do usuário✅ Sim⚠️ Não✅ Sim
Middle-tier com credencial✅ Sim✅ Sim⚠️ Não
Cenário multi-camada✅ IdealParcial⚠️ Não

Por que OBO com Graph API?

  • O usuário faz login na sua aplicação
  • O middle-tier precisa acessar SharePoint, OneDrive, Mail, Teams em nome desse usuário
  • Você não quer dar permissão de aplicativo para ler tudo do tenant
  • A auditoria no SharePoint mostra quem realmente acessou

📌 Nesta PoC: listamos documentos de uma biblioteca SharePoint que o usuário autenticado pode ver

Fluxo completo

Fase 1 · Autenticação do usuário
Navegador Usuário abre a aplicação em
https://localhost:7011
1 Redireciona para o login
Microsoft Entra ID Autentica o usuário e valida a identidade
2 Retorna a sessão autenticada
Web App Recebe o usuário autenticado e prepara o acesso downstream
Fase 2 · OBO + Microsoft Graph + SharePoint
Web App Solicita token delegado para o Graph usando certificado X.509
3 Executa o exchange OBO
Entra ID Emite token delegado para Graph em nome do usuário
4 Entrega o token downstream
Microsoft Graph Resolve site, biblioteca e consulta o conteúdo
5 Consulta SharePoint com identidade delegada
SharePoint Online Retorna apenas os documentos que o usuário pode acessar
Resultado esperado: a aplicação mostra os documentos da biblioteca SharePoint sem usar permissão de aplicativo ampla e sem expor tokens ao usuário.

Requisitos no Microsoft Entra ID

1. App Registration

Configure a redirect URI web:

https://localhost:7011/signin-oidc

2. Certificado X.509

Suba o .cer no app registration.

Evita client secret e melhora a postura de segurança.

3. Permissões delegadas

  • Sites.Read.All
  • Files.Read.All

4. Admin consent

Conceda consentimento administrativo se o tenant exigir.

Autenticação com certificado

Por que certificado?

  • Mais seguro que client secret
  • Não expira a cada 2 anos por default
  • Chave privada nunca sai da máquina
  • Só o .cer público vai para o Entra ID

Gerar na PoC

.\scripts\New-OboDemoCertificate.ps1 `
  -PfxPassword "SuaSenha!"

Gera o certificado no store local e exporta .cer + .pfx

Código: configuração da web app

// Program.cs — Configuração do middleware de autenticação

builder.Services
    .AddMicrosoftIdentityWebAppAuthentication(
        builder.Configuration, "AzureAd")
    .EnableTokenAcquisitionToCallDownstreamApi(
        builder.Configuration
            .GetSection("OboDemo:GraphScopes")
            .Get<string[]>() ?? Array.Empty<string>())
    .AddInMemoryTokenCaches();

EnableTokenAcquisitionToCallDownstreamApi habilita o OBO transparente.
O certificado é carregado automaticamente pela configuração AzureAd:ClientCertificates.

Código: aquisição do token OBO

// OboSharePointService.cs — Dentro de GetDocumentsAsync()

// 1. Pede token delegado para Graph em nome do usuário
var token = await _tokenAcquisition
    .GetAccessTokenForUserAsync(_options.GraphScopes);

// 2. Usa o token para chamar Graph
using var httpClient = _httpClientFactory.CreateClient("Graph");
httpClient.DefaultRequestHeaders.Authorization =
    new AuthenticationHeaderValue("Bearer", token);

// 3. Chama Graph para listar documentos do SharePoint
using var response = await httpClient
    .GetAsync(requestUri, cancellationToken);

O ITokenAcquisition do Microsoft.Identity.Web faz a troca OBO internamente.
O token resultante é delegado: o Graph vê a identidade do usuário, não da aplicação.

Código: resolução do alvo SharePoint

// O serviço resolve automaticamente a URL configurada

// Tenta localizar o site correto
GET /sites/{host}:{path}

// Localiza a biblioteca pelo nome
GET /sites/{siteId}/drives

// Lista os documentos da biblioteca ou pasta
GET /drives/{driveId}/root/children
    ?$top=10
    &$select=id,name,webUrl,lastModifiedDateTime,size,file,folder

URLs como https://tenant.sharepoint.com/BibliotecaXYZ são resolvidas automaticamente para site + drive + pasta.

Apêndice: FE + BE com OBO

Fase 1 · Front-end obtém token para a API
Front-end Usuário faz login e a aplicação pede um access token para a sua própria API
1 Autenticação e consentimento
Microsoft Entra ID Emite token com aud = API para o cliente autenticado
2 Retorna o token da API
Front-end Envia o bearer token para o backend no cabeçalho Authorization
Fase 2 · Backend executa o OBO para o Graph
Front-end Chama a API com o token recebido do Entra ID
3 Bearer token para a API
Backend / API Valida o token e solicita o OBO para obter um token downstream do Graph
4 Exchange OBO
Microsoft Entra ID Troca o token da API por um token delegado para o Microsoft Graph
5 Token do Graph
Microsoft Graph Recebe a chamada do backend em nome do usuário autenticado
Regra importante: o front-end envia para o backend um access token da API, não um token do Graph e não um id_token.
